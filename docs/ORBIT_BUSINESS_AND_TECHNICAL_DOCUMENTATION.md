## 1. Project Overview

### Project Name
ORBIT

### Tagline
Stay in your orbit. Keep progressing.

### Project Vision
ORBIT is a health-focused productivity platform that helps people build sustainable lifestyle momentum through daily tasks, focus sessions, proof-based missions, and community-driven gamification.

### Mission Statement
To turn healthy routines into a motivating, measurable, and social daily experience without making self-improvement feel clinical, boring, or isolating.

### Core Objectives
- Help users track hydration, sleep, movement, focus, and daily wellness habits.
- Convert personal progress into Health Points, streaks, badges, and visible milestones.
- Support small private groups through Orbits and large public communities through Galaxies.
- Encourage proof-based accountability through uploadable mission evidence.
- Build a scalable product foundation for consumer wellness, education, teams, and enterprise wellness programs.

### Problem The Platform Solves
Many users want better health and productivity, but they struggle with consistency. Existing habit, Pomodoro, and wellness apps often solve only one narrow problem, fail to provide meaningful community accountability, or feel like static trackers instead of motivating daily systems.

### Existing Market Gap
The market has strong single-purpose tools:
- Pomodoro and focus tools help with time blocking.
- Fitness and wellness apps track health metrics.
- Habit apps track completion.
- Community platforms create social belonging.

The gap is a unified experience that connects health, focus, proof-based missions, gamification, and small-group accountability in one coherent product.

### User Pain Points
- Lack of motivation after the first few days.
- Too many separate apps for focus, health, tasks, and communities.
- Static dashboards that do not feel rewarding.
- Habit tracking without emotional engagement.
- Public communities feel too broad; private accountability groups are missing or weak.
- Progress is often invisible, which reduces retention.
- Students and young professionals need lighter, friendlier systems than enterprise productivity software.

### Why Current Solutions Fail
- They track actions but do not create meaningful progression loops.
- They rely on notifications instead of identity, social accountability, and rewards.
- They separate productivity from wellness even though the user experiences both together.
- They often lack proof, trust, and challenge systems.
- Many have weak onboarding and poor personalization.

### Proposed Innovation
ORBIT combines:
- Health tracking
- Focus sessions
- Daily missions
- Proof upload
- HP rewards
- Streaks and badges
- Private Orbits
- Public Galaxies
- Future AI recommendations

The innovation is not just feature aggregation; it is the creation of a lifestyle progression engine.

### Competitive Advantage
- Private accountability groups using invitation codes and links.
- Public Galaxy challenges for global participation.
- Health and productivity connected under one gamified score.
- Proof-based missions to increase trust and accountability.
- Strong visual identity with a futuristic but friendly design system.
- Expandable architecture for AI personalization, school programs, wellness campaigns, and enterprise teams.

### Unique Selling Points
- One platform for health, focus, and missions.
- Community split between private Orbits and public Galaxies.
- Health Points as a unified motivation system.
- Mascot-driven emotional design.
- Proof uploads for real-world accountability.
- Future AI coach and recommendation engine.

## 3. User Roles and Permissions

### Guest
Responsibilities: Explore landing, register, login.  
Permissions: View public marketing pages.  
Restrictions: No access to dashboard or private data.  
Dashboard Features: None.  
Actions: Register, login, view invite link landing.  
Security Level: Public.  
Data Scope: Public content only.  
Logic: Redirect authenticated routes to login.

### Registered User
Responsibilities: Track habits, complete missions, join challenges.  
Permissions: Manage own profile, focus sessions, health logs, mission proofs.  
Restrictions: Cannot access other users' private data.  
Dashboard Features: Today overview, focus, health, missions, profile, community.  
Actions: Update profile, upload proof, join Orbit, join Galaxy challenges.  
Security Level: Authenticated.  
Data Scope: Own user document plus permitted community data.  
Logic: Firebase ID token verified by backend.

### Orbit Member
Responsibilities: Participate in private Orbit activity.  
Permissions: View Orbit summary, members, private challenges.  
Restrictions: Cannot manage Orbit settings unless owner/moderator.  
Dashboard Features: Orbit card, member activity, group challenges.  
Actions: Join challenges, nudge members, leave Orbit.  
Security Level: Authenticated plus membership check.  
Data Scope: Own Orbit only.  
Logic: Membership stored in Orbit members subcollection.

### Orbit Owner
Responsibilities: Create and manage a private Orbit.  
Permissions: Invite members, manage group metadata, start group challenges.  
Restrictions: Cannot access private user health details unless aggregated/consented.  
Dashboard Features: Orbit management, invite code/link, engagement overview.  
Actions: Create Orbit, remove abusive members, edit Orbit name.  
Security Level: Owner role.  
Data Scope: Owned Orbit metadata and aggregate activity.  
Logic: Owner UID on Orbit document.

### Orbit Moderator
Responsibilities: Keep private group safe and active.  
Permissions: Moderate posts/proofs/challenges depending on owner rules.  
Restrictions: Cannot delete Orbit or change billing.  
Dashboard Features: Reports, pending content, member moderation.  
Actions: Warn, remove content, resolve reports.  
Security Level: Elevated community role.  
Data Scope: Assigned Orbit.  
Logic: Role field in membership document.

### Galaxy Participant
Responsibilities: Join global challenges.  
Permissions: View public Galaxy challenges and own progress.  
Restrictions: Cannot alter public challenge definitions.  
Dashboard Features: Galaxy challenge cards, progress, leaderboard.  
Actions: Join, claim rewards, report content.  
Security Level: Authenticated.  
Data Scope: Public challenge data plus own progress.  
Logic: Challenge participant record per user.

### Admin
Responsibilities: Operate platform safely.  
Permissions: Manage users, challenges, moderation, platform settings.  
Restrictions: Sensitive actions require audit logs and possibly step-up auth.  
Dashboard Features: Users, reports, analytics, challenges, system health.  
Actions: Suspend accounts, create Galaxy challenges, resolve reports.  
Security Level: High.  
Data Scope: Platform-wide operational data.  
Logic: Custom claims or admin roles collection.

### Super Admin
Responsibilities: Own critical configuration and compliance.  
Permissions: Full administrative access.  
Restrictions: Cannot bypass audit logging.  
Dashboard Features: Security, billing, infrastructure, admin management.  
Actions: Assign admins, configure global settings, export compliance data.  
Security Level: Critical.  
Data Scope: Full platform scope.  
Logic: Hardware MFA and strict allowlist recommended.

### Enterprise Admin
Responsibilities: Manage organization workspace.  
Permissions: Invite employees/students, view aggregate analytics, manage campaigns.  
Restrictions: No invasive access to individual private health details unless explicit policy and consent.  
Dashboard Features: Organization analytics, campaigns, members, billing.  
Actions: Create org challenges, export aggregate reports.  
Security Level: Enterprise scoped.  
Data Scope: Organization users and aggregates.  
Logic: Tenant ID and org role checks.

## 4. Full Feature Breakdown

### Authentication
Purpose: Secure account access.  
User Benefit: Easy login with email and Google.  
Business Value: Trust and identity foundation.  
Technical Logic: Firebase Auth client obtains ID token; backend verifies token.  
Frontend: Login/register forms, auth provider, protected routes.  
Backend: Token verification middleware, profile creation.  
Database: User profile document.  
APIs: `/api/auth/me`, `/api/users/me`.  
Security: Token expiry, HTTPS, input validation.  
Edge Cases: Expired token, deleted account, email not verified.  
Scalability: Stateless auth verification.  
Performance: Cache profile response client-side.

### Profile Management
Purpose: User identity and progress summary.  
User Benefit: Personalized profile, username, avatar, badges.  
Business Value: Retention and social identity.  
Technical Logic: Backend profile service merges auth profile and stored profile.  
Frontend: Edit modal, avatar choices, timeline, rewards.  
Backend: PATCH profile validation.  
Database: Users collection.  
APIs: `/api/users/me`, `/api/users/rewards`, `/api/users/activity`.  
Security: User can edit only own profile.  
Edge Cases: Missing Google photo, invalid username, duplicate usernames.  
Scalability: Unique username index needed in relational migration.  
Performance: Avoid loading large activity history by default.

### Health Tracking
Purpose: Track hydration, sleep, movement, health score.  
User Benefit: Daily wellness awareness.  
Business Value: Core differentiation from productivity-only apps.  
Technical Logic: Daily health document per user/date.  
Frontend: Simple inputs, score visualization.  
Backend: Validate metrics and award HP once per daily state.  
Database: `users/{uid}/dailyHealth/{dateKey}`.  
APIs: `/api/health/daily`.  
Security: Authenticated writes only.  
Edge Cases: Timezone differences, repeated updates, impossible values.  
Scalability: Partition by user and date.  
Performance: Load only current day and recent history.

### Focus Sessions
Purpose: Pomodoro and deep work tracking.  
User Benefit: Structured focus plans and HP rewards.  
Business Value: Daily engagement loop.  
Technical Logic: Preset plans, timer completion, backend session write.  
Frontend: Preset cards, custom plan, pause/reset, complete only after timer ends.  
Backend: Store completed sessions and HP.  
Database: `users/{uid}/focusSessions`.  
APIs: `/api/focus/sessions`.  
Security: Prevent fake excessive rewards through server validation and rate limits.  
Edge Cases: Browser close, offline mode, clock manipulation.  
Scalability: High-write collection; eventually move to event stream/table.  
Performance: Avoid expensive composite indexes by sorting limited data in memory.

### Daily Missions
Purpose: Daily action list with proof upload.  
User Benefit: Clear daily goals.  
Business Value: Differentiated accountability.  
Technical Logic: Daily mission templates generated per user/day; proof stored externally.  
Frontend: Mission cards, upload modal, proof preview, completion state.  
Backend: Upload middleware, Cloudinary integration, mission completion.  
Database: `dailyMissions/{date}/missions`.  
APIs: `/api/missions/daily`, `/api/missions/:id/proof`.  
Security: File type/size validation, malware scanning at scale.  
Edge Cases: Upload fails, duplicate submission, large images.  
Scalability: CDN-backed object storage.  
Performance: Lazy load proofs.

### Gamification
Purpose: HP, streaks, badges, rewards.  
User Benefit: Visible progression.  
Business Value: Retention and monetization.  
Technical Logic: Reward events from focus, health, missions, challenges.  
Frontend: HP cards, badges, progress rings.  
Backend: Central reward ledger recommended.  
Database: Profile counters plus reward history.  
APIs: `/api/users/rewards`.  
Security: Server-authoritative HP.  
Edge Cases: Double claiming, race conditions.  
Scalability: Event sourcing or ledger table for auditability.  
Performance: Store aggregates for dashboard.

### Orbits
Purpose: Private small groups.  
User Benefit: Accountability with friends.  
Business Value: Viral invites and network effects.  
Technical Logic: Create Orbit, invite code/link, membership records.  
Frontend: Create/join UI, invite copy, member list.  
Backend: Orbit creation/join, code validation.  
Database: Global `orbits` collection and member subcollection.  
APIs: `/api/community/orbits`.  
Security: Membership authorization.  
Edge Cases: Duplicate join, full group, invalid code.  
Scalability: Shard large membership lists or move to relational joins.  
Performance: Store aggregate member count.

### Galaxies
Purpose: Public large communities and global challenges.  
User Benefit: Bigger motivation and events.  
Business Value: Sponsorship and growth channel.  
Technical Logic: Public challenges and leaderboards.  
Frontend: Challenge list, detail modal, progress, claim.  
Backend: Public challenge definitions and user progress calculation.  
Database: Challenges, participants, activity aggregates.  
APIs: `/api/community/summary`, `/api/community/challenges/join`, `/api/community/challenges/claim`.  
Security: Anti-cheat, moderation, rate limiting.  
Edge Cases: Leaderboard abuse, timezone boundaries.  
Scalability: Precomputed leaderboards.  
Performance: Cache public challenge data.

### Notifications
Purpose: Timely reminders and social updates.  
User Benefit: Better consistency.  
Business Value: Retention.  
Technical Logic: Event-triggered and scheduled notifications.  
Frontend: Notification center and preferences.  
Backend: Queue workers and notification service.  
Database: Notification table/collection.  
APIs: `/api/notifications`.  
Security: User opt-in and unsubscribe controls.  
Edge Cases: Notification fatigue.  
Scalability: Queue-based delivery.  
Performance: Batch fan-out.

### AI Coach
Purpose: Personalized recommendations and planning.  
User Benefit: Better next action suggestions.  
Business Value: Premium subscription differentiator.  
Technical Logic: Summarize recent activity, generate safe recommendations.  
Frontend: Coach panel, recommended plan card.  
Backend: AI orchestration service with safety filters.  
Database: User preferences, AI interaction logs, embeddings.  
APIs: `/api/ai/recommendations`, `/api/ai/chat`.  
Security: No medical diagnosis; privacy filtering.  
Edge Cases: Unsafe advice, hallucination, over-personalization.  
Scalability: Caching, batching, cheaper models for simple tasks.  
Performance: Async generation for weekly plans.

### Admin and Moderation
Purpose: Keep platform safe and manageable.  
User Benefit: Trustworthy communities.  
Business Value: Required for scale and enterprise.  
Technical Logic: Admin roles, reports, audit logs.  
Frontend: Admin dashboard.  
Backend: Admin middleware and moderation APIs.  
Database: Reports, audit logs, moderation decisions.  
APIs: `/api/admin/*`.  
Security: MFA, custom claims, audit trails.  
Edge Cases: Admin abuse, false reports.  
Scalability: Queue moderation and AI assist.  
Performance: Paginated admin lists.

## 5. User Flows

### User Registration
1. User opens landing page.
2. User clicks Get Started.
3. User chooses Google or email.
4. Firebase creates identity.
5. Frontend receives ID token.
6. Backend verifies token.
7. Backend creates profile if missing.
8. User lands on onboarding/dashboard.

### Authentication
1. User submits credentials or Google sign-in.
2. Firebase validates identity.
3. Frontend stores auth state in memory/Firebase SDK.
4. API requests attach ID token.
5. Backend `requireAuth` verifies token.
6. Backend returns scoped data.

### Password Recovery
1. User clicks Forgot Password.
2. Enters email.
3. Firebase sends reset email.
4. User resets password.
5. User logs in again.
6. Backend profile remains linked by UID.

### Onboarding
1. Ask user goal: study, work, health, balanced.
2. Ask preferred focus plan.
3. Ask hydration/sleep baseline.
4. Suggest first daily mission.
5. Offer Orbit creation or join code.
6. Show dashboard with today's plan.

### Main User Journey
1. Open dashboard.
2. See today's focus, health, mission, and community challenge.
3. Start focus session.
4. Log health progress.
5. Complete mission and upload proof.
6. Earn HP and badges.
7. Check Orbit/Galaxy progress.
8. Return next day through streak and notifications.

### Navigation Flow
Primary routes:
- Home
- Dashboard
- Focus
- Health
- Missions
- Community
- Profile
- Settings
- Admin, role-based

### Dashboard Flow
1. Load profile and daily summaries.
2. Display next best action.
3. Display HP, streaks, completed missions.
4. Show active community challenge.
5. Provide quick actions to focus, log health, complete mission.

### Admin Workflow
1. Admin signs in with elevated account.
2. Backend validates admin role.
3. Admin views reports and metrics.
4. Admin resolves content/user reports.
5. System records audit event.

### Payment Flow
1. User chooses plan.
2. Backend creates checkout session.
3. Payment provider handles checkout.
4. Webhook updates subscription status.
5. Frontend unlocks premium features.

### Notification Flow
1. User opts in.
2. Preferences stored.
3. Events trigger notifications.
4. Queue sends push/email/in-app message.
5. User can mute or adjust frequency.

### Error Handling Flow
1. Frontend catches API error.
2. Shows human-friendly message.
3. Logs technical detail to monitoring.
4. Backend returns structured error code.
5. Retriable actions provide retry state.

### AI Interaction Flow
1. User asks coach for plan.
2. Backend fetches recent activity.
3. Safety layer removes sensitive unnecessary data.
4. AI generates recommendation.
5. Backend validates output format.
6. User receives actionable plan.

## 7. Database Architecture

### MVP Database
Current MVP can continue on Firestore because it supports fast iteration, Firebase Auth integration, and flexible documents.

### Recommended Scale Database
For millions of users, migrate core transactional data to PostgreSQL while keeping object storage/CDN for media and a warehouse for analytics.

### Core Tables

#### users
- id UUID primary key
- firebase_uid varchar unique
- email varchar unique
- display_name varchar
- username varchar unique
- photo_url text
- hp integer default 0
- streak_count integer default 0
- created_at timestamptz
- updated_at timestamptz
- last_seen_at timestamptz

#### focus_sessions
- id UUID primary key
- user_id UUID foreign key users.id
- plan_key varchar
- plan_name varchar
- minutes integer
- hp integer
- completed_at timestamptz
- date_key date

Indexes: `(user_id, completed_at desc)`, `(user_id, date_key)`.

#### daily_health
- id UUID primary key
- user_id UUID foreign key users.id
- date_key date
- hydration_cups integer
- sleep_hours numeric
- movement_minutes integer
- score integer
- hp_awarded integer
- updated_at timestamptz

Constraint: unique `(user_id, date_key)`.

#### daily_missions
- id UUID primary key
- user_id UUID foreign key users.id
- date_key date
- template_key varchar
- title varchar
- detail text
- hp integer
- status varchar
- proof_url text
- completed_at timestamptz

Indexes: `(user_id, date_key)`, `(user_id, completed_at desc)`.

#### orbits
- id UUID primary key
- owner_id UUID foreign key users.id
- name varchar
- invite_code varchar unique
- member_count integer
- created_at timestamptz
- updated_at timestamptz

#### orbit_members
- orbit_id UUID foreign key orbits.id
- user_id UUID foreign key users.id
- role varchar
- joined_at timestamptz
- primary key `(orbit_id, user_id)`

#### challenges
- id UUID primary key
- scope varchar
- title varchar
- description text
- type varchar
- target integer
- reward_hp integer
- starts_at timestamptz
- ends_at timestamptz

#### challenge_participants
- challenge_id UUID foreign key challenges.id
- user_id UUID foreign key users.id
- joined_at timestamptz
- claimed_at timestamptz
- progress_snapshot integer

- id UUID primary key

- entity_type varchar
- entity_id varchar

### Relationships
- User has many focus sessions, health logs, missions, rewards, notifications.
- User belongs to zero or one personal Orbit in MVP; future can support multiple.
- Orbit has many members.
- Challenge has many participants.
- Reward ledger records all HP mutations.

### Normalization Logic
Core transactional data should be normalized. Dashboard aggregates can be denormalized into summary tables or materialized views for speed.

### Indexing Strategy
- User/date indexes for daily data.
- User/completed_at indexes for activity timelines.
- Invite code unique index.
- Challenge scope/status index.
- Reward ledger uniqueness for anti-duplication.

### Query Optimization
- Use pagination.
- Avoid unbounded subcollection reads.
- Precompute dashboard summaries.
- Cache public Galaxy challenge definitions.
- Use read replicas for analytics at scale.

### Backup Strategy
- Daily automated database backups.
- Point-in-time recovery.
- Separate object storage backup lifecycle.
- Regular restore drills.
- Audit log retention policy.

## 11. Security System

### Authentication Security
- Firebase Auth for MVP.
- Enforce verified email for sensitive actions.
- Google login supported.
- MFA for admins.

### JWT Strategy
- Frontend obtains Firebase ID token.
- Backend verifies token per request.
- Do not store long-lived tokens in localStorage manually.
- Use HTTPS only in production.

### Session Handling
- Short-lived ID tokens.
- Refresh handled by Firebase SDK.
- Backend remains stateless.
- Optional secure HTTP-only session cookies for web enterprise version.

### Encryption
- TLS in transit.
- Managed encryption at rest.
- Hash internal secrets.
- Use KMS for production secrets.

### Access Control
- RBAC for admin/moderator/org roles.
- Ownership checks for user records.
- Membership checks for Orbits.
- Tenant isolation for enterprise.

### Common Attack Prevention
- SQL injection: parameterized queries and ORM/query builder.
- XSS: React escaping, strict CSP, sanitize user content.
- CSRF: SameSite cookies or token headers if cookie sessions are used.
- File attacks: MIME sniffing, extension validation, size limits, scanning.
- DDoS: CDN/WAF/rate limiting.

### Privacy
- Collect minimum health data.
- Clear privacy policy.
- Data export/delete.
- Consent for analytics and notifications.
- Aggregate-only organization reports by default.

### Audit Logs
Required for:
- Admin actions.
- Role changes.
- Account suspensions.
- Reward corrections.
- Enterprise exports.

## 16. Final Technical Recommendations

### Best Stack
Frontend: Next.js, React, TypeScript, Tailwind, Framer Motion.  
Backend: TypeScript API, Express now, Fastify/NestJS later if structure grows.  
Database: Firestore for MVP, PostgreSQL for scale-critical core data.  
Auth: Firebase Auth now; enterprise SSO later.  
Storage: Cloudinary now; object storage/CDN strategy at scale.  
AI: Backend-managed AI service with structured outputs and safety layer.  
Hosting: Vercel frontend, Cloud Run/Fly/Render backend, Cloudflare for CDN/WAF.

### Why This Architecture Is Optimal
It balances speed and production readiness. Firebase and Cloudinary accelerate MVP development, while the domain boundaries already allow future migration to relational data, queues, analytics, and AI services.

### Potential Bottlenecks
- Firestore read costs for dashboards and community summaries.
- Challenge progress calculation if done live for large communities.
- File upload abuse.
- AI cost growth.
- Notification fan-out.
- Leaderboard computation.

### Cost Considerations
- Cache public data.
- Store aggregates.
- Limit proof image size.
- Use AI asynchronously and selectively.
- Use lifecycle rules for media.
- Monitor Firestore reads per active user.

### Future Scaling Challenges
- Multi-Orbit membership.
- Enterprise tenant isolation.
- Moderation at scale.
- Accurate anti-cheat logic.
- Data privacy compliance.
- Migrating from document-first MVP to relational/event architecture.

### Maintainability Strategy
- Keep feature modules isolated.
- Add shared API schemas.
- Add tests around reward and challenge logic.
- Introduce repository/data access layer before database migration.
- Use design tokens and reusable UI primitives.
- Document every domain decision.
